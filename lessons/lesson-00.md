## 📚 핵심 개념

### 1. Git과 GitHub은 완전히 다른 것
| 구분 | Git | GitHub |
|---|---|---|
| 정체 | **프로그램** (내 PC에 설치) | **웹사이트** (인터넷 서비스) |
| 역할 | 파일 변경 이력을 기록하는 타임머신 | Git 저장소를 인터넷에 백업/공유 |
| 위치 | 로컬 (내 컴퓨터) | 원격 (Microsoft 서버) |
| 비유 | CCTV 현장의 NVR (녹화 장치) | 본사 서버실 (중앙 백업) |

> 📝 **NVR** (Network Video Recorder): IP 카메라의 영상을 네트워크로 받아 녹화하는 장치. CCTV 현장에서 흔히 보는 녹화기.

### 2. 왜 Git을 사용하는가?
- **시간 여행**: 언제든 과거 시점의 코드로 되돌아갈 수 있음
- **변경 추적**: 누가, 언제, 무엇을, 왜 바꿨는지 기록됨
- **협업 가능**: 여러 명이 같은 프로젝트를 동시에 작업 가능
- **백업**: GitHub에 푸시하면 내 PC가 망가져도 코드 보존
- **포트폴리오**: 18개월간의 학습 여정이 공개 기록으로 남음

### 3. 파일 저장(Ctrl+S) vs 커밋의 차이 ⭐
- **파일 저장 (Ctrl+S)**: 현재 상태를 디스크에 덮어씀 → 과거 상태는 사라짐
- **커밋 (commit)**: 현재 상태를 Git의 역사책에 "한 페이지"로 남김 → 과거 상태 영구 보존

**CCTV 비유**: 화면을 덮어쓰는 방식 vs 매 순간을 타임스탬프와 함께 따로따로 저장하는 방식. Git은 후자.

## 🔑 필수 용어 사전

### Git 기본 개념
| 용어 | 영문 | 뜻 |
|---|---|---|
| **저장소** | Repository (Repo) | Git이 감시하는 프로젝트 폴더 |
| **로컬 저장소** | Local Repository | 내 PC에 있는 저장소 |
| **원격 저장소** | Remote Repository | GitHub 같은 인터넷에 있는 저장소 |
| **작업 디렉토리** | Working Directory | 실제로 파일을 수정하는 폴더 |
| **스테이징 영역** | Staging Area (Index) | 커밋 직전 파일들을 모아두는 대기실 |
| **커밋** | Commit | 현재 상태를 영구 기록으로 남기는 것 (스냅샷) |
| **커밋 메시지** | Commit Message | 왜 이렇게 바꿨는지 설명하는 메모 |
| **해시** | Hash | 각 커밋마다 붙는 고유 식별 번호 (예: `a3f5e2b...`) |
| **HEAD** | HEAD | 현재 작업 중인 커밋을 가리키는 포인터 |

### 파일의 3단계 상태 (매우 중요)
Git에서 파일은 항상 세 가지 상태 중 하나에 있다:

```
[ 작업 디렉토리 ]  →  [ 스테이징 영역 ]  →  [ 저장소 ]
  (수정된 파일)      (커밋 대기 파일)      (커밋 완료)
        │                  │                    │
        │   git add        │   git commit       │
        └──────────────────┴────────────────────┘
```

1. **Modified (수정됨)**: 파일을 수정했지만 아직 Git에 알리지 않은 상태
2. **Staged (스테이징됨)**: `git add`로 "이거 커밋할 거야"라고 표시한 상태
3. **Committed (커밋됨)**: `git commit`으로 역사책에 기록된 상태

**비유**: 택배 보내기 과정
- Modified = 집에서 물건 포장 중
- Staged = 포장 끝내고 택배 박스에 넣음 (아직 안 보냄)
- Committed = 택배 기사에게 전달 완료 (기록 남음)

### Git 핵심 명령어
| 명령어 | 의미 | 언제 사용 |
|---|---|---|
| `git init` | 현재 폴더를 Git 저장소로 만들기 | 새 프로젝트 시작할 때 |
| `git clone [URL]` | 원격 저장소를 내 PC로 통째로 복사 | 남의 프로젝트 가져올 때 |
| `git status` | 현재 파일 상태 확인 | **가장 자주 쓰는 명령어** |
| `git add [파일명]` | 특정 파일을 스테이징 영역에 올리기 | 커밋 준비할 때 |
| `git add .` | 변경된 모든 파일을 스테이징 | 한꺼번에 올릴 때 |
| `git commit -m "메시지"` | 스테이징된 파일들을 커밋 | 작업 단락 지을 때 |
| `git push` | 로컬 커밋을 원격 저장소로 업로드 | GitHub에 올릴 때 |
| `git pull` | 원격 저장소의 변경사항을 로컬로 가져오기 | 다른 사람 작업 반영할 때 |
| `git fetch` | 원격 변경사항 확인만 (병합 X) | 확인만 하고 싶을 때 |
| `git log` | 커밋 기록 보기 | 과거 기록 확인할 때 |
| `git diff` | 변경된 내용 보기 | 뭐가 바뀌었는지 확인할 때 |

### 원격 저장소 관련
| 용어 | 뜻 |
|---|---|
| **origin** | 원격 저장소의 기본 이름 (관례적으로 GitHub 주소를 가리킴) |
| **push** | 로컬 → 원격 (올리기) |
| **pull** | 원격 → 로컬 (내리기) + 병합 자동 실행 |
| **fetch** | 원격 → 로컬 (내리기만, 병합 X) |
| **clone** | 원격 저장소를 처음 복사해올 때 |

### 브랜치 (아직 안 배웠지만 알아두기)
| 용어 | 뜻 |
|---|---|
| **브랜치** (Branch) | 작업을 나눠서 하기 위한 "가지". 메인 코드와 분리된 작업 공간 |
| **main** | 기본 브랜치 이름 (옛날엔 master였음) |
| **체크아웃** (Checkout) | 다른 브랜치로 이동 |
| **머지** (Merge) | 두 브랜치를 합치기 |
| **풀 리퀘스트** (Pull Request, PR) | "내가 만든 브랜치를 main에 합쳐줘"라고 요청 |
| **컨플릭트** (Conflict) | 두 브랜치를 합칠 때 같은 부분이 다르게 수정되어 충돌 |

> 📌 브랜치는 8개월 차에 자세히 배울 예정. 지금은 혼자 공부하니까 main 브랜치 하나만 사용.

### 기타 중요 용어
| 용어 | 뜻 |
|---|---|
| **.gitignore** | Git이 무시할 파일/폴더 목록을 적는 파일 |
| **README.md** | 저장소 소개 파일 (저장소 첫 화면에 표시됨) |
| **이슈** (Issue) | GitHub에서 버그나 할 일을 기록하는 기능 |
| **포크** (Fork) | 남의 저장소를 내 계정으로 복사 |
| **스타** (Star) | 저장소 즐겨찾기 (북마크 같은 것) |
| **릴리스** (Release) | 프로젝트의 정식 배포 버전 |
| **태그** (Tag) | 특정 커밋에 이름표 붙이기 (주로 버전 표시용) |

## 📝 커밋 메시지 규칙 (본 저장소 규칙)

모든 커밋은 아래 4가지 접두사 중 하나로 시작한다.

| 접두사 | 언제 쓰나 | 예시 |
|---|---|---|
| `feat:` | 새 기능 추가 | `feat: 번호판 검증 함수 추가` |
| `fix:` | 버그 수정 | `fix: 소수점 계산 오류 수정` |
| `docs:` | 문서 작성/수정 | `docs: README 작성` |
| `study:` | 공부 기록 | `study: Lesson 3 디버깅 예제 작성` |

### 좋은 커밋 메시지 원칙
1. **접두사 필수**: `feat:`, `fix:`, `docs:`, `study:` 중 하나로 시작
2. **현재 시제**: "추가했음" ❌ → "추가" ✅
3. **구체적**: "수정" ❌ → "소수점 반올림 오류 수정" ✅
4. **간결하게**: 한 줄로, 50자 이내 권장

## 🛠️ 초기 설정 명령어 (한 번만 실행)

Git을 처음 설치하면 "내가 누구인지" 알려줘야 한다.

```bash
# 사용자 이름 설정
git config --global user.name "내 이름 또는 닉네임"

# 이메일 설정 (GitHub 계정과 동일해야 함)
git config --global user.email "내 이메일"

# 설정 확인
git config --list
```

### Git Bash 사용 팁
| 작업 | 단축키 |
|---|---|
| 복사 | `Ctrl + Insert` 또는 드래그 후 우클릭 |
| 붙여넣기 | `Shift + Insert` 또는 우클릭 → Paste |
| 자동완성 | `Tab` |
| 이전 명령 | `↑` 방향키 |

> ⚠️ `Ctrl + C`, `Ctrl + V`는 Git Bash에서 작동하지 않는다!

## ⚠️ 주의사항 및 함정

### 절대 하지 말 것
1. **비밀번호, API 키, 개인정보를 커밋하지 말 것**
   → Public 저장소면 전 세계에 공개됨
   → 5개월 차 Lesson 207~208에서 User Secrets 배울 예정
2. **`bin/`, `obj/`, `.vs/` 폴더를 커밋하지 말 것**
   → `.gitignore`로 자동 제외됨 (Lesson 11-1에서 자세히)
3. **커밋 메시지를 "수정", "업데이트" 같이 애매하게 쓰지 말 것**

### 자주 겪는 문제
- **"push가 안 돼요"**: 먼저 `git pull`로 원격 변경사항 받아온 후 다시 push
- **"커밋을 취소하고 싶어요"**: `git reset HEAD~1` (마지막 커밋만 되돌리기)
- **"파일을 실수로 추가했어요"**: `git reset HEAD [파일명]` (스테이징 취소)
