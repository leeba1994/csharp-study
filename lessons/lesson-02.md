## 📚 핵심 개념

### 1. 솔루션 vs 프로젝트 (입문자가 90% 헷갈리는 개념)

| 용어 | 정의 | 파일 확장자 | 비유 |
|---|---|---|---|
| **솔루션 (Solution)** | 관련된 여러 프로젝트를 묶는 그릇 | `.sln` | 오피스 빌딩 |
| **프로젝트 (Project)** | 실제 프로그램 하나를 만드는 단위 | `.csproj` | 빌딩 안의 회사 하나 |

**핵심 규칙**:
- 실제 실행 파일(`.exe`)이 만들어지는 건 **프로젝트 단위**
- 솔루션은 프로젝트들을 모아두는 상위 개념일 뿐
- 코드는 모두 **프로젝트 안**에 있음

### 2. 왜 솔루션/프로젝트로 나누나?

대형 프로그램은 역할별로 여러 프로젝트로 분리하는 것이 실무 표준이다.

**예시**: 18개월 후 완성될 번호판 OCR 관제 프로그램
```
📦 LicensePlateOCR.sln (솔루션 = 전체 프로그램)
   ├─ 🧩 LicensePlateOCR.UI          (WPF 관제 화면 - 실행 파일)
   ├─ 🧩 LicensePlateOCR.Core        (OCR 로직 - 라이브러리)
   ├─ 🧩 LicensePlateOCR.Data        (DB 처리 - 라이브러리)
   ├─ 🧩 LicensePlateOCR.Network     (네트워크 통신 - 라이브러리)
   └─ 🧩 LicensePlateOCR.Tests       (테스트 코드)
```

**분리의 장점**:
1. **유지보수**: 한 기능만 고치고 싶을 때 다른 파트 안 건드림
2. **재사용**: 라이브러리 프로젝트를 다른 프로그램에서도 사용 가능
3. **협업**: 팀원별로 다른 프로젝트 담당 가능
4. **테스트**: 테스트 코드와 실제 코드 분리

### 3. 콘솔 앱이란?

- **콘솔(Console)**: 검정 바탕에 글자만 나오는 창 (cmd, Git Bash 같은 것)
- **콘솔 프로그램**: 버튼/이미지 없이 글자로 입력받고 글자로 출력하는 가장 단순한 프로그램
- **왜 콘솔부터?**: 화려한 UI 걱정 없이 C# 언어 자체에만 집중하기 위함
- **학습 로드맵**:
  - Phase 1 (1~5개월): 콘솔로 C# 언어 완전 정복
  - Phase 2 (6개월~): WPF로 창 있는 프로그램

### 4. 프로젝트 생성 시 핵심 선택 사항

#### ① 템플릿 선택: "콘솔 앱" (C#, .NET)
**고를 것**:
- ✅ **콘솔 앱** (C#, Linux/macOS/Windows 태그 포함)

**피해야 할 함정**:
- ❌ 콘솔 앱 **(.NET Framework)** — 옛날 Windows 전용 구버전
- ❌ 콘솔 앱 (Visual Basic) — 다른 언어
- ❌ 콘솔 앱 (F#) — 다른 언어

**체크 포인트**:
- 태그에 `C#` 있는가?
- 태그에 `Linux, macOS` 도 함께 있는가? (= 통합 .NET 8)
- 이름 끝에 `(.NET Framework)` 같은 괄호가 없는가?

#### ② 프로젝트 이름 규칙
- ✅ 영어만 사용 (한글 절대 금지)
- ✅ **PascalCase** (각 단어 첫 글자 대문자): `HelloCSharp`, `LicensePlateOCR`
- ✅ 공백 없이 연결

> ⚠️ 한글 이름 쓰면 Git에서 깨지거나 빌드 에러 발생 가능

#### ③ 위치 (매우 중요)
- **반드시 Git 저장소 폴더 안에** 생성: `D:\dev\csharp-study\`
- 이래야 Git이 추적하고 GitHub에 푸시됨

#### ④ "최상위 문 사용 안 함" 체크박스

**✅ 반드시 체크!**

| 체크 안 함 (A 스타일, 기본값) | ✅ 체크함 (B 스타일, 전통 방식) |
|---|---|
| C# 9 (2020)에 추가된 단축형 | C# 1 (2002)부터의 전통 방식 |
| 한 줄로 작성 가능 | 기본 11줄 구조 |
| 간단 테스트용 | **실무 표준 (90%)** |
| 회사 코드 거의 없음 | **회사 코드 100%** |
| 대형 프로젝트 부적합 | 대형 프로젝트 적합 |
| 개념들이 숨겨져 있음 | namespace/class/Main/static 등 개념 드러남 |

**왜 체크하는가?**:
1. 학습 목적으로는 B 스타일이 압도적으로 좋음
2. 회사 코드는 100% B 스타일
3. B 스타일의 각 "껍데기"는 중요한 개념을 드러냄
4. A는 뒷면에서 결국 B로 번역됨 (A는 축약형일 뿐)

### 5. 프로젝트 생성 후 폴더 구조

```
📦 csharp-study\                    ← Git 저장소 (이미 존재)
   └─ 📁 HelloCSharp\                ← 솔루션 폴더 (새로 생성)
       ├─ 📄 HelloCSharp.sln         ← 솔루션 파일
       └─ 📁 HelloCSharp\             ← 프로젝트 폴더 (새로 생성)
           ├─ 📄 HelloCSharp.csproj   ← 프로젝트 파일
           └─ 📄 Program.cs           ← 실제 C# 코드 파일
```

같은 이름의 폴더가 두 번 겹쳐 보이지만, **솔루션과 프로젝트가 각자 자기 폴더를 가지는 정상 구조**이다.

### 6. 첫 C# 코드 구조 (Program.cs)

```csharp
namespace HelloCSharp        // 네임스페이스 (껍데기 1)
{
    internal class Program   // 클래스 (껍데기 2)
    {
        static void Main(string[] args)  // Main 메서드 (껍데기 3) - 프로그램 시작점
        {
            Console.WriteLine("Hello, World!");  // 실제 명령 (이 한 줄이 진짜 일함)
        }
    }
}
```

**지금 단계에서 알아야 할 3가지**:

1. **대부분이 "껍데기"**: 11줄 중 실제 명령은 1줄뿐. 나머지는 C# 문법 규칙
2. **중괄호 `{` `}` 쌍**: 항상 여는 것과 닫는 것이 짝이 맞아야 함
3. **세미콜론 `;`**: 명령 한 줄 끝에 반드시 붙여야 함

> 껍데기들(namespace, class, Main, static)의 정확한 의미는 5개월 차까지 천천히 배움.
> 지금은 "있어야 하는 것들"로만 알아두면 됨.

### 7. Visual Studio 화면 구조

```
┌──────────────────────────────────┬─────────────────┐
│  📝 코드 편집 영역 (Program.cs)    │  🗂️ 솔루션 탐색기 │
│                                   │                  │
│  C# 코드 작성/수정                 │  프로젝트 구조   │
├──────────────────────────────────┴─────────────────┤
│  📤 출력 창 (Output)                                │
│  빌드 결과, 에러 메시지 표시                         │
└────────────────────────────────────────────────────┘
```

### 8. 프로그램 실행 (F5)

**F5 키** = 디버그 모드로 실행 (가장 많이 쓰는 단축키)

F5 누르면 자동으로 일어나는 일:
```
1. 코드 검사: 문법 오류 확인
    ↓
2. 컴파일: C# 코드 → 중간 언어(IL)로 번역
    ↓
3. 빌드: HelloCSharp.exe 파일 생성
    ↓
4. 실행: Windows가 .exe를 메모리에 로드
    ↓
5. .NET 런타임 시작: Main 메서드 찾기
    ↓
6. Console.WriteLine 실행
    ↓
7. 콘솔 창에 출력
    ↓
8. Main 종료 → 프로그램 종료 → 종료 코드 0 반환
```

**종료 코드의 의미**:
- `0` = 정상 종료 ✅
- `1`, `-1` 등 = 오류 발생

## ⚠️ 오늘 겪은 돌발 상황 3건 (해결 기록)

### 1. NAS 경로 Git Ownership 에러
- **증상**: `fatal: detected dubious ownership in repository`
- **원인**: 저장소가 NAS(네트워크 드라이브)에 있어서 Git이 소유권 확인 불가
- **해결**: `git clone`으로 로컬 `D:\dev\`에 새로 받음
- **교훈**: 개발 작업은 항상 **로컬 SSD/HDD**에서 진행. NAS는 부적합.

### 2. GitHub 인증 에러
- **증상**: `Invalid username or token. Password authentication is not supported`
- **원인**: GitHub가 2021년 8월부터 비밀번호 인증 중단. 토큰 또는 SSH 필요.
- **해결**: `git push origin main` 실행 시 Git Credential Manager(GCM)가 자동으로 브라우저 인증 처리
- **교훈**: Windows Git은 GCM이 자동 내장되어 있어 편리함

### 3. NuGet NU1302 에러 (HTTP 원본 불가)
- **증상**: `error NU1302: 'HTTP' 원본을 사용하여 'restore' 작업... NuGet에 HTTPS 원본이 필요합니다`
- **원인**: 회사 PC에 등록된 `nuget.grapecity.com`이 HTTP 방식. 최신 .NET은 HTTPS만 허용.
- **해결**: 도구 > 옵션 > NuGet 패키지 관리자 > 패키지 소스 > GrapeCity 체크 해제
- **교훈**: 학습 프로젝트는 외부 라이브러리 필요 없음. 문제 되는 소스는 비활성화 OK.

## 💡 "최상위 문 사용 안 함" 더 깊이 이해하기

### 두 스타일 비교 (같은 "Hello, World!")

**스타일 A (최상위 문, 단축형)**:
```csharp
Console.WriteLine("Hello, World!");
```
→ 1줄. 끝.

**스타일 B (전통 방식, 우리 선택)**:
```csharp
namespace HelloCSharp
{
    internal class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("Hello, World!");
        }
    }
}
```
→ 11줄.

### 어떤 코드에 어떤 스타일?

- **스타일 A**: 간단한 스크립트, 데모 코드, 5분짜리 실험
- **스타일 B**: **실무 코드, 대형 프로젝트, 학습용** (우리 선택)

### 왜 B를 선택했나?

1. **코드가 길어지면 A 스타일로는 이해하기 어려움**
2. 회사 개발팀의 지반침하/안티드론 프로그램 코드 = **100% B 스타일**
3. B 스타일은 C#의 핵심 개념(namespace, class, Main, static)을 자연스럽게 노출
4. 어차피 컴파일러가 A를 내부적으로 B로 번역함 (A는 B의 축약형)

> 핵심 원칙: **"편해 보이는 것 ≠ 학습에 좋은 것"**
